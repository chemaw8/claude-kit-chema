# Kit Chema — Plan de implementación

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Construir el Kit Chema v1.0: paquete distribuible (núcleo CLAUDE.md + 7 skills + hooks + instalador + guías + deck) que hace que Claude produzca output de alta calidad consistente para cualquier persona de Innovattia.

**Architecture:** Repo de contenido en `~/Trabajo/proyectos/claude-kit-chema/`. Un núcleo corto que carga siempre, playbooks como skills que cargan bajo demanda, reglas duras como hooks deterministas, un instalador bash no destructivo y verificación automatizada de límites (`verificar.sh`). Spec: `docs/superpowers/specs/2026-07-08-claude-kit-design.md`.

**Tech Stack:** Markdown (formato Agent Skills de Claude Code), Bash (instalador, hook, verificador), Python3 (solo dentro de scripts bash para JSON), skill `pptx` para el deck.

## Global Constraints

- Todo el contenido en español; sin MAYÚSCULAS de énfasis ni "CRITICAL:"; redacción concreta y verificable (spec P5).
- `nucleo/CLAUDE.md` < 150 líneas (spec P1).
- Cada `description` de skill < 1024 caracteres, con QUÉ + CUÁNDO + frases gatillo literales (spec P4).
- Cada `SKILL.md` < 5.000 palabras (ideal < 1.500); detalle largo va en `references/` (spec P4).
- Nombres de skills con prefijo `kit-` (evita colisiones con skills ya instaladas): `kit-presentaciones`, `kit-analisis-datos`, `kit-research`, `kit-codigo`, `kit-propuestas`, `kit-finanzas`, `kit-automatizacion`.
- Marcadores del instalador (texto exacto): `<!-- kit-chema:inicio -->` y `<!-- kit-chema:fin -->`.
- Archivos y carpetas en kebab-case sin acentos; fechas absolutas.
- Commits frecuentes, mensajes en español, con `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>`.
- Lo crítico al inicio de cada archivo (spec P6). Nada de placeholders "TBD".

### Task 1: Andamiaje del repo + verificador de límites

**Files:**
- Create: `README.md`, `CHANGELOG.md`, `verificar.sh`

**Interfaces:**
- Produces: `./verificar.sh` — sale con código 0 si todos los límites pasan, 1 si no; imprime una línea `OK`/`FALLA` por chequeo. Las tareas 2-15 lo usan como test.

- [ ] **Step 1: Escribir `verificar.sh` (el "test" del kit)**

```bash
#!/usr/bin/env bash
# Verifica los límites cuantitativos del Kit Chema (spec P1, P4).
set -uo pipefail
cd "$(dirname "${BASH_SOURCE[0]}")"
fallas=0

chk() { # chk <descripcion> <condicion(0=ok)>
  if [ "$2" -eq 0 ]; then echo "OK    $1"; else echo "FALLA $1"; fallas=1; fi
}

# 1. Núcleo < 150 líneas
if [ -f nucleo/CLAUDE.md ]; then
  lineas=$(wc -l < nucleo/CLAUDE.md)
  [ "$lineas" -lt 150 ]; chk "nucleo/CLAUDE.md tiene $lineas líneas (< 150)" $?
fi

# 2. Descriptions < 1024 caracteres y con frases gatillo
for f in skills/*/SKILL.md; do
  [ -e "$f" ] || continue
  desc=$(awk '/^description:/{sub(/^description:[ ]*/,""); print; exit}' "$f")
  n=${#desc}
  [ "$n" -gt 0 ] && [ "$n" -lt 1024 ]; chk "$f description $n chars (0<n<1024)" $?
done

# 3. SKILL.md < 5000 palabras
for f in skills/*/SKILL.md; do
  [ -e "$f" ] || continue
  palabras=$(wc -w < "$f")
  [ "$palabras" -lt 5000 ]; chk "$f tiene $palabras palabras (< 5000)" $?
done

# 4. Sin gritos: mayúsculas de énfasis prohibidas en contenido instalable
if grep -rnE '(CRITICAL|IMPORTANTE:|OBLIGATORIO:|NUNCA HAGAS|SIEMPRE DEBES)' nucleo/ skills/ 2>/dev/null | grep -v ':#'; then
  chk "sin énfasis gritado en nucleo/ y skills/" 1
else
  chk "sin énfasis gritado en nucleo/ y skills/" 0
fi

exit $fallas
```

**Files:**
- Create: `nucleo/CLAUDE.md`

**Interfaces:**
- Produces: los nombres de skills referenciados en la tabla final (`kit-*`, ver Global Constraints) y las palabras clave `council`, `rápido`, `hazlo directo` que COMO-PEDIR.md (Task 13) documenta idénticas.

- [ ] **Step 1: Escribir el núcleo completo (borrador aprobado en spec 5.1; este es el texto base, ajustar solo si `verificar.sh` falla)**

```markdown
# Kit Chema — estándar de calidad para Claude

Este archivo define cómo se trabaja en cualquier conversación. Las reglas por
dominio viven en las skills del kit (tabla al final).

## Contexto de empresa

Antes de un trabajo sustantivo, lee `~/.claude/contexto/CONTEXTO-EMPRESA.md`
(audiencia, tono, glosario, confidencialidad). Si no existe, pregunta al
usuario lo mínimo: para quién es el trabajo y qué es confidencial.

## Arranque de tarea

Antes de empezar un trabajo sustantivo confirma que conoces: propósito,
audiencia, formato esperado y criterio de éxito. Si falta algo crítico,
pregunta — máximo 2-3 preguntas, las de mayor impacto. Para tareas triviales
(consultas rápidas, ediciones menores) responde directo, sin protocolo.

## Evaluación crítica

Cuando el usuario proponga una idea, decisión o solución, evalúala antes de
ejecutarla y responde en este orden: qué está bien, qué me preocupa (con
evidencia concreta), qué haría yo. Ejecuta sin evaluar solo si el usuario
dice "hazlo directo". Detectar un error en la propuesta del usuario y
callarlo es el peor fallo posible bajo este manual.

## Esfuerzo proporcional al riesgo

- Tarea trivial → respuesta directa.
- Entregable estándar → aplica el playbook del dominio y su checklist.
- Decisión cara o irreversible, o material que sale de la empresa → council
  (ver skill kit-propuestas). La palabra "council" lo fuerza; "rápido" lo
  salta bajo responsabilidad del usuario.

## Terminado significa verificado

Nada se declara listo sin comprobarlo: código ejecutado, cifras recalculadas,
fuentes abiertas y citadas, archivos generados abiertos y revisados. Reporta
lo que falló o quedó fuera; un "listo" falso cuesta más que un "me faltó
esto".

## Cero residuos

Al cerrar una tarea elimina archivos temporales, pruebas sueltas y código
muerto. El resultado final solo contiene lo que aporta.

## Continuidad entre sesiones

- Trabajo a medias → deja `CONTINUAR.md` en la carpeta del trabajo: estado,
  siguiente paso y cómo retomar.
- Decisión importante → anótala en `DECISIONES.md` del proyecto: fecha, qué
  se decidió, razones y alternativas descartadas. Consúltalo antes de
  reabrir discusiones cerradas.
- Fechas absolutas siempre (2026-07-08, no "ayer").

## Modelos para subagentes

Al delegar en subagentes o workflows: Haiku para lo trivial, Sonnet para lo
mecánico (buscar, extraer, verificaciones tipo checklist), Opus 4.8 para
trabajo pesado intermedio (borradores, evaluadores de council, código),
Fable 5 solo para síntesis final y juicio crítico.

## Mejora del kit

Si el usuario corrige dos veces lo mismo, propón convertir esa corrección en
regla del kit (repo claude-kit-chema, con entrada en su CHANGELOG).

## Playbooks por dominio

| Si la tarea es… | Usa la skill |
|---|---|
| Presentación, deck, informe con audiencia | kit-presentaciones |
| Análisis o exploración de datos | kit-analisis-datos |
| Investigación con fuentes | kit-research |
| Código, scripts, desarrollo | kit-codigo |
| Propuesta o decisión a evaluar | kit-propuestas |
| Números, proyecciones, dinero | kit-finanzas |
| Automatizar un proceso | kit-automatizacion |
```

name: kit-codigo
description: Estándar Kit Chema para escribir, modificar o revisar código, scripts y proyectos de desarrollo. Úsala siempre antes de tocar código — al crear una función o proyecto, arreglar un bug, hacer un script, refactorizar o revisar un PR. Cuerpo (< 1.500 palabras), secciones con este contenido (spec 5.2):
1. **Bien hecho significa**: funciona verificado end-to-end (no solo tests); sigue los patrones del repo; cambio mínimo que resuelve la tarea; sin residuos; sin secretos ni datos reales de clientes; commits atómicos con mensajes claros.
2. **Proceso**: entender (leer el código existente relevante antes de escribir) → plan corto → TDD cuando hay lógica que probar (test que falla → implementación mínima → test pasa) → verificación end-to-end real (correr la app/script con datos de verdad) → limpieza del diff.
3. **Higiene anti-contaminación** (la sección más importante): antes de crear una función/módulo, buscar si ya existe algo que lo haga (grep primero, escribir después); respetar estilo y naming del repo aunque no te guste; nada de refactors no pedidos; borrar código muerto, prints de debug y archivos de prueba antes de terminar; toda dependencia nueva se justifica en una línea (y se verifica que no haya ya una equivalente); revisar el diff completo antes del commit final: lo que no aporta, fuera.
4. **Debugging sistemático**: reproducir el error antes de arreglarlo; hipótesis → evidencia → fix; nunca "arreglar" lo que no se entiende.
5. **Checklist final**: ¿corre de verdad? ¿tests pasan? ¿diff limpio? ¿sin secretos? ¿commits atómicos? ¿documenté cómo correrlo si es nuevo?
6. **Errores típicos**: duplicar utilidades existentes; "mejoras" no pedidas; declarar listo sin ejecutar; dejar archivos-experimento en el repo.

### Task 5: Skill `kit-propuestas` (con protocolo Council)

**Files:**
- Create: `skills/kit-propuestas/SKILL.md`

**Interfaces:**
- Produces: nombre `kit-propuestas`; el protocolo Council que el núcleo dispara con la palabra "council". Veredictos exactos: `aprobada` / `aprobada con cambios` / `rechazada`.

- [ ] **Step 1: Escribir SKILL.md**

Frontmatter exacto:

```yaml
---
name: kit-propuestas
description: Estándar Kit Chema para redactar o evaluar propuestas y decisiones — propuestas de desarrollo, de inversión, de arquitectura, cotizaciones, o cualquier "¿hacemos X o Y?". Úsala al escribir una propuesta, al evaluar una idea del usuario con consecuencias reales, y siempre que se pida "council". Frases gatillo "hazme la propuesta", "¿te parece bien esta idea?", "evalúa esta decisión", "council". Incluye el protocolo Council: panel de evaluadores independientes con veredicto aprobada / con cambios / rechazada.
---
```

Cuerpo (< 1.500 palabras):
1. **Estructura de toda propuesta**: problema (qué duele y a quién) → opciones (2-3 reales, no espantapájaros) → costos (dinero, tiempo, riesgo) por opción → recomendación con razones → qué haría falta para revertirla.
2. **Cuándo corre un Council**: (a) el usuario dice "council"; (b) decisión cara o irreversible; (c) material que sale de la empresa. Si no aplica, basta la evaluación crítica del núcleo.
3. **Protocolo Council**: 3-5 evaluadores independientes de contexto fresco, cada uno con un lente: viabilidad técnica, costo/beneficio, riesgos, abogado del diablo, y opcionalmente el dominio relevante. En Claude Code: subagentes en paralelo con modelo Opus 4.8, veredicto sintetizado en el hilo principal. En claude.ai web: evaluar secuencialmente con cada lente en la misma conversación, marcando cada sección.
4. **Mandato acotado del evaluador** (texto que se pasa a cada evaluador): "Reporta solo hallazgos que cambien la corrección o la decisión, cada uno con evidencia concreta. 'Sin hallazgos relevantes' es una respuesta válida y esperada. No rellenes por cumplir."
5. **Veredicto**: `aprobada` / `aprobada con cambios` (lista concreta) / `rechazada` (razones); si el council no puede decidir, decir qué evidencia falta. Antes de opinar, consultar `DECISIONES.md` del proyecto para no reabrir lo decidido.
6. **Errores típicos**: council teatral (inventar fallos para parecer riguroso); opciones de paja; recomendar sin costos; reabrir decisiones cerradas.

#!/usr/bin/env bash
# Hook PreToolUse (Bash) del Kit Chema: bloquea git commit/push si el diff
# staged contiene patrones de credenciales. Exit 2 = bloquear (stderr al modelo).
set -uo pipefail
input=$(cat)
cmd=$(printf '%s' "$input" | python3 -c 'import json,sys; print(json.load(sys.stdin).get("tool_input",{}).get("command",""))' 2>/dev/null || echo "")
case "$cmd" in
  *"git commit"*|*"git push"*) ;;
  *) exit 0 ;;
esac
patron='(api[_-]?key|secret|token|passw(or)?d)["'"'"']?[[:space:]]*[:=][[:space:]]*["'"'"']?[A-Za-z0-9_/+.-]{12,}|AKIA[0-9A-Z]{16}|sk-[A-Za-z0-9_-]{20,}|ghp_[A-Za-z0-9]{30,}|-----BEGIN [A-Z ]*PRIVATE KEY-----'
if git diff --cached 2>/dev/null | grep -qE "$patron"; then
  echo "Kit Chema: el diff staged parece contener credenciales (clave API, token o llave privada). Revísalo y quítalas antes de commitear; si es un falso positivo, dilo y commitea con el hook desactivado." >&2
  exit 2
fi
exit 0

### Task 12: Instalador (`instalar.sh`)

**Files:**
- Create: `instalar.sh`

**Interfaces:**
- Consumes: `nucleo/CLAUDE.md`, `skills/kit-*`, `contexto/`, `hooks/` (Tasks 2-11); marcadores exactos de Global Constraints.
- Produces: instalación en `$CLAUDE_DIR` (default `~/.claude`), variable respetada para tests.

- [ ] **Step 1: Escribir `instalar.sh`**

```bash
#!/usr/bin/env bash
# Instalador del Kit Chema. No destructivo: nunca pisa contenido ajeno.
set -euo pipefail
KIT="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
DIR="${CLAUDE_DIR:-$HOME/.claude}"
INI="<!-- kit-chema:inicio -->"
FIN="<!-- kit-chema:fin -->"
VER=$(grep -m1 -oE '## v[0-9]+\.[0-9]+' "$KIT/CHANGELOG.md" | cut -d' ' -f2)

mkdir -p "$DIR" "$DIR/skills" "$DIR/contexto" "$DIR/hooks"

# 1. Núcleo dentro de marcadores
bloque=$(printf '%s\n<!-- Kit Chema %s — no editar dentro de los marcadores; se actualiza con instalar.sh -->\n%s\n%s' "$INI" "$VER" "$(cat "$KIT/nucleo/CLAUDE.md")" "$FIN")
destino="$DIR/CLAUDE.md"
if [ ! -f "$destino" ]; then
  printf '%s\n' "$bloque" > "$destino"
  echo "núcleo: instalado (nuevo $destino)"
elif grep -qF "$INI" "$destino"; then
  awk -v ini="$INI" -v fin="$FIN" -v bloque="$bloque" '
    $0==ini {dentro=1; print bloque; next}
    $0==fin {dentro=0; next}
    !dentro {print}' "$destino" > "$destino.tmp" && mv "$destino.tmp" "$destino"
  echo "núcleo: sección kit-chema actualizada a $VER"
else
  printf '\n%s\n' "$bloque" >> "$destino"
  echo "núcleo: añadido al final de tu CLAUDE.md existente (no se tocó tu contenido)"
fi

# 2. Skills (solo las del kit; sobrescribe únicamente kit-*)
for d in "$KIT"/skills/kit-*/; do
  nombre=$(basename "$d")
  rm -rf "$DIR/skills/$nombre"
  cp -r "$d" "$DIR/skills/$nombre"
  echo "skill: $nombre"
done

# 3. Contexto: solo si no existe (es del usuario)
if [ ! -f "$DIR/contexto/CONTEXTO-EMPRESA.md" ]; then
  cp "$KIT/contexto/CONTEXTO-EMPRESA.md" "$DIR/contexto/"
  echo "contexto: plantilla instalada — edítala con tus datos"
else
  echo "contexto: ya existe, no se toca"
fi

# 4. Hooks: opt-in
resp="${KIT_HOOKS:-}"
if [ -z "$resp" ] && [ -t 0 ]; then
  read -r -p "¿Instalar hook anti-secretos (bloquea commits con credenciales)? [s/N] " resp
fi
if [ "${resp,,}" = "s" ]; then
  cp "$KIT/hooks/anti-secretos.sh" "$DIR/hooks/" && chmod +x "$DIR/hooks/anti-secretos.sh"
  python3 - "$DIR/settings.json" "$KIT/hooks/settings-fragment.json" <<'PY'
import json, sys, os
destino, fragmento = sys.argv[1], sys.argv[2]
s = json.load(open(destino)) if os.path.exists(destino) else {}
f = json.load(open(fragmento))
hooks = s.setdefault("hooks", {})
for evento, entradas in f["hooks"].items():
    actuales = hooks.setdefault(evento, [])
    for e in entradas:
        if e not in actuales:
            actuales.append(e)
json.dump(s, open(destino, "w"), indent=2, ensure_ascii=False)
PY
  echo "hooks: anti-secretos activado"
else
  echo "hooks: omitidos (KIT_HOOKS=s ./instalar.sh para activarlos)"
fi

echo
echo "Kit Chema $VER instalado en $DIR. Verifica: abre una conversación nueva y pide un deck — Claude debe preguntar audiencia y objetivo."
```
